# BIDL Language Specification

The Bharat Interface Definition Language (BIDL) provides a C-like, strongly-typed syntax for defining IPC and uRPC contracts. This document outlines the minimal, phase-1 grammar and semantics.

## 1. File Structure and Versioning

Every `.bidl` file defines a single logical scope (typically a service contract) and must declare an explicit semantic version at the top.

```bidl
// namesvc.bidl
version 1.0;

namespace bharat.namesvc;
```

## 2. Primitive Data Types

BIDL strongly encourages explicit integer sizing. It avoids ambiguous C types (e.g., `int`, `long`).

* **Scalars:** `u8`, `u16`, `u32`, `u64`, `i8`, `i16`, `i32`, `i64`, `f32`, `f64`, `bool`.

### 2.1 Bounded Types
To preserve real-time predictability and bounded buffer allocation, unbounded data is **forbidden**. Strings and dynamic arrays must specify a strict upper bound.

* **Bounded Strings:** `string<MaxLen>`
  ```bidl
  string<64> service_name;
  ```
* **Bounded Arrays:** `Array<Type, MaxLen>`
  ```bidl
  Array<u32, 16> ip_addresses;
  ```

## 3. Handles & Capabilities

A core tenet of BIDL is capability-awareness. A "handle" represents an opaque token or object capability passed across IPC boundaries.

* **Handles:** `handle<ResourceType>`

```bidl
handle<endpoint> reply_port;
handle<memory_region> shared_buffer;
```

## 4. Enums and Flags

Enums provide named constants, defaulting to `u32` but supporting explicit underlying types.

```bidl
enum Status : i32 {
    OK = 0,
    ERR_NOT_FOUND = -1,
    ERR_DENIED = -2,
    ERR_INVALID = -3
}
```

Bitflags provide bitwise OR-able constants, explicitly sized.

```bidl
flags OpenMode : u32 {
    READ = 0x01,
    WRITE = 0x02,
    EXEC = 0x04
}
```

## 5. Structs

Structs group data fields. They are deterministic, meaning the BIDL compiler controls padding to ensure predictable ABI layouts.

```bidl
struct EndpointInfo {
    string<64> name;
    u32 protocol_version;
    handle<endpoint> connection_port;
}
```

## 6. Interfaces and Methods

An `interface` defines a named collection of operations (`methods`). Interfaces map to C dispatch tables or uRPC event demuxers.

```bidl
interface NameService {
    // A standard synchronous method
    method resolve(string<64> name) -> (handle<endpoint> port, Status status);

    // A method requiring specific capability rights to invoke
    @requires(capability="NET_ADMIN", rights="WRITE")
    method register(string<64> name, handle<endpoint> port) -> (Status status);
}
```

### 6.1 Method Semantics
* `method name(Args) -> (Returns);`
* Arguments are typically passed in the request message payload.
* Returns are returned in the response message payload.

### 6.2 Events (Asynchronous)
For one-way notifications or uRPC streams, `event` is used instead of `method`. Events have no response payload.

```bidl
interface NetworkEvents {
    event link_state_changed(u32 interface_id, bool is_up);
}
```

## 7. Annotations

Annotations influence how the BIDL compiler generates runtime glue code.

### 7.1 Transport Annotations
By default, the runtime determines transport based on connection type. However, a developer can force a specific generation target:

* `@transport(endpoint)`: Enforces generation of synchronous local IPC structures and `sys_send`/`sys_recv` stubs.
* `@transport(urpc)`: Enforces generation of uRPC ring-buffer packet layouts and async event queue stubs.

```bidl
@transport(endpoint)
interface LocalDevMgr { ... }
```

### 7.2 Capability Annotations
The `@requires` annotation maps operations to object-based access control. The dispatcher generated by BIDL will perform these checks *before* calling the actual service implementation.

```bidl
@requires(capability="SYS_ADMIN", rights="EXECUTE")
method shutdown() -> (Status status);
```

## 8. Canonical Error Model

While custom enums like `Status` can be used, BIDL strongly encourages a canonical error type mapped to the kernel's `kstatus_t` to avoid status code translation boundaries.

```bidl
// Implicitly mapped to internal K_ERR_* constants
method query_stats(u32 id) -> (Status status, DevStats stats);
```
