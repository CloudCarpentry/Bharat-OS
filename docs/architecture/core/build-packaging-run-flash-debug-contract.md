# Build, Packaging, Run, Flash, and Debug Contract

This document outlines the manifest-driven target delivery pipeline architecture for Bharat-OS. The pipeline transforms declarative inputs (YAML targets or legacy JSON configs) into runnable or flashable artifacts while strictly decoupling the compilation, packaging, and execution phases.

**Note on Orchestration:** As per the memory core rules, the unified Python build orchestrator is `tools/build.py`. Execution relies on explicit subcommands (e.g., `build`, `package`, `run`, `flash`, `debug`, `all`) and explicit flags: `--target-yaml <path>` for YAML targets and `--target <name>` for legacy configurations. Do not guess behavior from positional arguments.

## 1. Architectural Boundaries

The build-to-run pipeline enforces rigid boundaries between systems:

- **Build (`tools/build/build_executor.py`)**: Responsible only for invoking CMake and producing canonical artifacts (like `kernel.elf`). It has no knowledge of how to format binaries for specific emulators or flashers.
- **Package (`tools/package/packager.py`)**: Derives target-specific artifacts (like `raw_bin` or `flash_image`) from canonical build outputs through registered transforms. It creates deterministic execution plans.
- **Run (`tools/run/`)**: Consumes a resolved, packaged run manifest. It does not parse original YAML targets or read from the build directory. It expects fully formed paths to packaged run artifacts.
- **Flash/Debug (`tools/flash/`, `tools/debug/`)**: Consumes resolved, packaged manifests to communicate with external hardware or debugger stubs.

## 2. Artifact Types

There is a hard distinction between artifact types:

### Canonical Artifacts
The direct output of the build system (CMake/Ninja).
- Examples: `kernel.elf`, `kernel.map`, symbol files.
- These are universal and not specific to any bootloader or runner requirements.

### Derived Artifacts
Outputs generated by the packaging layer by applying transforms to canonical artifacts.
- Examples: `kernel.bin` (via objcopy), `boot.img` (via mkimage).
- These are customized to meet specific boot protocol or hardware requirements.

### Run Artifact vs Flash Artifact
- A **run artifact** is passed to an emulator (e.g., `-kernel kernel.bin` for QEMU).
- A **flash artifact** is written to physical ROM or flash memory (e.g., an OpenOCD payload).
- These may be the same underlying file, or they may diverge entirely depending on the target profile.

## 3. The Manifest Generation Rule

Orchestration logic happens in stages, and state is passed between stages via JSON manifests.

- `build-manifest.json`: Emitted by the build executor. Records canonical artifact paths and toolchain information.
- `run-manifest.json`: Emitted by the packager. Specifies the execution backend (e.g., `qemu`), machine configs, and the exact path to the `boot_artifact`.
- `flash-manifest.json`: Emitted by the packager. Specifies the flash backend and the artifact path to program.
- `debug-manifest.json`: Emitted by the packager. Contains paths to symbol files and target connection parameters.

**Rule:** Execution layers (Run, Flash, Debug) *only* read their respective JSON manifests. They never parse the original target YAML or legacy JSON configs.

## 4. Boot Protocol and DTB (Device Tree Blob) Ownership

Boot protocols dictate the expected format of the payload and the mechanism for passing hardware descriptions (like a DTB).

- **Boot Protocols**: Dictate how the environment hands off to the kernel (e.g., `linux_arm64`, `multiboot2`, `raw_entry`).
- **Artifact Formats**: Dictate the shape of the binary (e.g., `elf`, `raw_bin`).

### DTB Handling Strategies
- `qemu_generated`: The emulator constructs and passes the DTB in memory. The package layer does not produce a DTB.
- `appended`: The package layer appends the compiled DTB directly to the kernel binary.
- `external`: The DTB exists as a separate file and is passed explicitly by the runner/flasher.

## 5. Transition Plan: `build_config.json` to YAML Targets

The legacy `build_config.json` is deprecated. However, to ensure a smooth transition without fork-lifting the entire architecture at once:

1. Both YAML targets and legacy configurations are read by a single `target_resolver` module.
2. Both inputs are normalized into a unified `ResolvedTarget` internal Python model before any pipeline logic runs.
3. The orchestration pipeline only understands the `ResolvedTarget` model, ensuring unified logic for both legacy and modern inputs.
4. Legacy inputs will gracefully map to default "fallback" pipeline behaviors, but new features (like complex flash topologies or advanced emulation configs) are only supported via YAML targets.
