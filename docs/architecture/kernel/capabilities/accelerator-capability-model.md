# Accelerator Capability Model

## 1. The Missing Model for Complex Devices

In early Bharat-OS designs, hardware access was often modeled using generic `CAP_TYPE_DEVICE_MMIO` capabilities. While this suffices for simple MMIO registers, it is fundamentally flawed for modern, complex accelerators like GPUs, NPUs, and advanced DMA controllers.

These devices manage their own address spaces, submit queues, telemetry counters, and internal partitions. Treating them simply as "memory regions" fails to isolate disparate workloads sharing the same physical hardware.

To address this, Bharat-OS introduces explicit, first-class capability families specifically for accelerator subsystems.

## 2. Capability Families for Accelerators

The following capability types represent distinct, granular authorities over an accelerator's resources:

### `CAP_TYPE_ACCEL_DEVICE`
- **Purpose:** Represents authority over the entire physical device. This is the root capability from which all other accelerator capabilities are derived.
- **Holders:** Typically held only by the core platform device manager or a highly privileged device service.

### `CAP_TYPE_ACCEL_QUEUE`
- **Purpose:** Represents authority to submit work to a specific hardware queue or execution context on the device.
- **Holders:** Delegated to tenant workloads (e.g., an ML application) that need to execute tasks.

### `CAP_TYPE_ACCEL_BUFFER`
- **Purpose:** Represents authority over a specific region of memory mapped into the accelerator's address space. It is distinct from host CPU memory.
- **Holders:** Delegated to workloads to provide input data or receive output data.

### `CAP_TYPE_ACCEL_TELEMETRY`
- **Purpose:** Represents authority to read performance counters, utilization metrics, and fault registers for the device or a specific queue.
- **Holders:** Delegated to monitoring services or the AI Governor for scheduling decisions.

### `CAP_TYPE_ACCEL_ADMIN`
- **Purpose:** Represents privileged administrative authority over the device, such as resetting the hardware, loading firmware, or repartitioning resources.
- **Holders:** Restricted strictly to supervisor or administration services.

### `CAP_TYPE_DMA_DOMAIN`
- **Purpose:** Represents an IOMMU protection domain associated with the accelerator.
- **Holders:** Held by the kernel or memory manager to enforce isolation.

### `CAP_TYPE_DMA_GRANT`
- **Purpose:** Represents a temporal lease (mapping) of host physical memory into the `CAP_TYPE_DMA_DOMAIN`. (See `dma-grant-model.md`).

## 3. Granular Rights

Each capability family has a specific set of rights that can be attenuated during delegation.

| Capability Family | Available Rights | Example Operations |
| :--- | :--- | :--- |
| **ACCEL_DEVICE** | `ALL`, `DERIVE` | Retyping to Queues or Admin capabilities. |
| **ACCEL_QUEUE** | `ENQUEUE`, `CANCEL`, `QUERY` | Submitting a job descriptor; aborting a pending job. |
| **ACCEL_BUFFER** | `MAP`, `BIND`, `SHARE`, `SYNC_CPU`, `SYNC_DEV` | Mapping into host VSpace; flushing caches to/from the device. |
| **ACCEL_TELEMETRY** | `READ_STATS`, `READ_FAULTS` | Polling utilization metrics; reading error codes. |
| **ACCEL_ADMIN** | `RESET`, `PARTITION`, `FW_LOAD`, `THROTTLE_OVERRIDE` | Hard-resetting the NPU; updating microcode. |
| **DMA_GRANT** | `MAP`, `UNMAP`, `REVOKE` | Binding host memory to an IOMMU IOVA. |

## 4. The Critical Rule: No Direct "Device Access"

In the Bharat-OS accelerator model, untrusted tenant workloads **NEVER** receive direct MMIO access (`CAP_TYPE_DEVICE_MMIO`) to an accelerator.

Instead, a tenant workload interacts with the accelerator exclusively through:
1.  **A Queue Capability (`ACCEL_QUEUE`):** To submit jobs.
2.  **A Buffer Capability (`ACCEL_BUFFER`):** Or a DMA Grant Lease, to provide data.

The actual device service (e.g., `services/device/accelmgr/`) holds the `ACCEL_DEVICE` capability, multiplexes the physical hardware, and enforces the capability contracts generated by BIDL. This ensures that a malicious workload cannot reprogram the device to bypass IOMMU protections or access another tenant's data.
