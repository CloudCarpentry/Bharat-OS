# CPU Affinity & NUMA

## Overview
In multi-core systems, especially Non-Uniform Memory Access (NUMA) architectures, the physical location of a thread (`bh_thread_t`) significantly impacts its performance. Bharat-OS allows explicit control over where threads execute.

## CPU Affinity (Pinning)
CPU Affinity is the property that determines which physical CPU cores a thread is allowed to run on.

-   **Mechanism:** Every thread has a bitmask (e.g., a 64-bit integer on a 64-core system, or a larger array for more cores) representing the allowed CPUs.
-   **Default:** When a task is created, its threads typically inherit the affinity mask of the parent task (often all available CPUs).
-   **System Call:** A thread can use a capability-authorized system call (e.g., `sys_set_affinity`) to restrict its execution to a specific subset of cores.

### Use Cases for Pinning
1.  **Real-Time (RT) Isolation:** A critical motor-control thread can be pinned to Core 3. All other non-RT threads can have Core 3 removed from their affinity masks. This guarantees the RT thread never competes for L1/L2 cache or CPU time on that core.
2.  **Cache Locality (GP):** If a thread frequently accesses a specific memory buffer, pinning it to one core ensures the data remains hot in that core's L1/L2 cache, avoiding costly cache invalidations across the interconnect.
3.  **Heterogeneous Computing (big.LITTLE / P-Cores & E-Cores):** Threads can be pinned to high-performance cores for demanding tasks or efficiency cores for background tasks. The AI Governor often manages this dynamically via `AI_ACTION_MIGRATE_TASK`.

## NUMA Awareness
In NUMA systems (common in multi-socket servers or chiplet designs), memory is divided into "nodes". A CPU core can access memory attached to its local node much faster than memory attached to a remote node.

### The Bharat-OS Approach
Bharat-OS integrates NUMA awareness deeply into its Physical Memory Manager (PMM) and Scheduler.

1.  **Memory Allocation:** When a thread allocates physical memory (via an Untyped capability), the kernel's PMM preferentially allocates pages from the NUMA node local to the CPU core the thread is currently running on.
2.  **Scheduling:** The scheduler attempts to keep a thread on the same NUMA node where its memory was allocated.
3.  **Migration Cost:** If the load balancer or the AI Governor decides to migrate a thread to balance CPU usage, it heavily weighs the cost of moving the thread across NUMA boundaries. Migrating a thread means its previously local memory accesses will now be remote and slow.

### Multikernel NUMA
Because Bharat-OS is a multikernel, NUMA awareness is implicit in the design.
-   **No Shared State:** Cores do not share a global runqueue or global page tables. They communicate via explicit URPC messages.
-   **Data Gravity:** If Core 0 (Node A) needs a large dataset generated by Core 16 (Node B), it is often faster for Core 0 to send a URPC request to Core 16 to process the data locally and return the small result, rather than migrating the thread and the data across the slow NUMA interconnect.
